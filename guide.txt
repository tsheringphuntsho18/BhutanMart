Assignment: Designing a Production-Ready E-Commerce Backend with MongoDB and Redis

Module Code: DBS302
Assignment Weightage: 25 Marks (25% of total module marks)
Type: Individual / Team-based Project Assignment
Mode of Submission: Source Code Repository + Technical Report + Demonstration
1. Assignment Overview

In this assignment, you are required to design, implement, and document the backend data layer of a production-ready e-commerce platform using two complementary NoSQL technologies: MongoDB as the primary document store and Redis as the in-memory key-value store. The objective is to demonstrate your ability to make informed architectural decisions, model data effectively across two distinct NoSQL paradigms, and apply database concepts to satisfy real-world functional and non-functional requirements.

You are free to use any frontend framework (React, Angular, Vue, Next.js, plain HTML/CSS, etc.) and any backend language or framework (Node.js/Express, Python/FastAPI/Django, Java/Spring Boot, Go, .NET, etc.) of your choice. The frontend and backend stack will not be the primary focus of evaluation. Marks will be awarded based on how appropriately and effectively MongoDB and Redis are applied to the problem domain.
2. Learning Outcomes

On successful completion of this assignment, the student will be able to:

    LO1: Design and implement schema patterns suitable for a document-oriented database (MongoDB) including embedded documents, references, and polymorphic structures.
    LO2: Apply Redis data structures (strings, hashes, lists, sets, sorted sets, HyperLogLog, etc.) to solve domain-specific problems such as caching, session management, and real-time analytics.
    LO3: Justify the selection of a NoSQL technology against functional and non-functional requirements based on the CAP theorem and consistency models.
    LO4: Implement and reason about indexing, query optimization, replication, sharding, and persistence in MongoDB.
    LO5: Apply caching strategies, eviction policies, and high-availability configurations in Redis.
    LO6: Address production-grade concerns such as security, scalability, performance, and observability in NoSQL deployments.

3. Problem Statement

You are hired as a backend engineer at "XYZ Shope" (change name accordinly), a fast-growing online retail startup. The company is moving away from its legacy relational database due to scalability bottlenecks and is adopting a polyglot persistence strategy. Your job is to design and build the data layer of the platform such that it can handle:

    A growing product catalogue with diverse and evolving attributes.
    High-throughput read traffic during flash sales and promotional events.
    Real-time features such as cart management, session handling, recommendations, and leaderboards.
    Reliable order processing with strong consistency guarantees.
    Analytical workloads such as trending products, daily active users, and inventory reports.

You must justify every architectural choice you make in your technical report.
4. Functional Requirements

Your application must support the following functional capabilities. The exact UI implementation is not graded; only the data layer behaviour is.
4.1 User Management

    User registration, login, and authentication (token-based, e.g., JWT).
    User profile management with addresses, payment preferences, and wishlists.
    Role-based access (Customer, Seller, Administrator).

4.2 Product Catalogue

    CRUD operations on products with categories, sub-categories, and variants (size, colour, etc.).
    Support for product attributes that vary across categories (e.g., "RAM" for laptops, "fabric" for clothing).
    Full-text search on product names, descriptions, and tags.
    Filtering, sorting, and pagination of product listings.

4.3 Shopping Cart and Sessions

    Add to cart, update quantity, remove from cart.
    Cart persistence across sessions for logged-in users.
    Guest cart support for non-authenticated users.
    Session expiration and TTL-based cleanup.

4.4 Order Processing

    Place an order with multiple line items.
    Order status workflow: Placed → Confirmed → Shipped → Delivered / Cancelled / Returned.
    Atomic stock decrement on order placement.
    Order history retrieval by user.

4.5 Real-Time Features (Redis-driven)

    Display of "Top 10 Trending Products" updated in real-time.
    Recently viewed products per user.
    Rate limiting on API endpoints (e.g., login attempts, checkout requests).
    Leaderboard of top sellers / top buyers of the month.
    Counting unique product page visits (using HyperLogLog).

4.6 Analytics and Reporting

    Daily and monthly sales reports using MongoDB's aggregation framework.
    Low-stock alert reports.
    Most viewed vs most purchased product analysis.

5. Non-Functional Requirements

You are expected to address the following non-functional concerns and document the steps you have taken to meet them.
# 	Concern 	Expectation
NFR1 	Performance 	Hot read paths (product details, homepage, cart) must be served from Redis cache. Document cache-hit ratios.
NFR2 	Scalability 	Provide a sharding plan for MongoDB collections. Identify shard keys with justification.
NFR3 	High Availability 	Configure a MongoDB replica set (minimum 3 nodes) and Redis Sentinel or Redis Cluster for failover.
NFR4 	Consistency 	Justify your read/write concerns in MongoDB. Explain trade-offs against CAP theorem.
NFR5 	Durability 	Configure Redis persistence (RDB, AOF, or hybrid) with justification.
NFR6 	Security 	Enable authentication, role-based access, and TLS in both databases. Never store plaintext passwords.
NFR7 	Observability 	Provide logging and basic monitoring (e.g., slow query logs, Redis INFO statistics).
NFR8 	Data Integrity 	Apply MongoDB multi-document transactions where business logic demands atomicity (e.g., order placement, stock update).
6. Specific Database Design Requirements

These requirements are central to your evaluation. Your submission must demonstrate explicit and thoughtful use of both databases.
6.1 MongoDB Requirements

    Design at least six collections (e.g., users, products, orders, categories, reviews, inventory) with appropriate schema.
    Demonstrate the use of embedding vs referencing with justification for each major collection.
    Create at least four indexes, including one compound index and one text index. Document why each was created.
    Implement at least two aggregation pipelines for analytical queries (e.g., monthly revenue, top-N products).
    Use multi-document ACID transactions for at least one critical workflow (order placement is recommended).
    Configure a replica set with three nodes (can be done locally with Docker or via MongoDB Atlas).
    Provide a sharding plan (theoretical design is acceptable if local sharding is not feasible). Identify candidate shard keys and justify the choice.
    Discuss your chosen read concern and write concern levels for critical operations.

6.2 Redis Requirements

    Use at least four distinct Redis data types (e.g., String, Hash, List, Set, Sorted Set, HyperLogLog, Bitmap). Each usage must be justified by the use case.
    Implement a caching layer for product details with appropriate TTL and a clearly defined cache-invalidation strategy.
    Implement session management for logged-in users using Redis with expiration.
    Implement a trending products leaderboard using a Sorted Set.
    Implement rate limiting on at least one endpoint using Redis counters with expiry.
    Use HyperLogLog to estimate unique visitors per product.
    Configure persistence (RDB, AOF, or hybrid) and justify your choice.
    Demonstrate either Redis Sentinel or Redis Cluster for high availability (a documented configuration with screenshots is acceptable).
    Apply an appropriate eviction policy (e.g., allkeys-lru, volatile-ttl) and justify the choice.

6.3 Cross-Database Concerns

    Define and document a cache-aside (lazy loading), write-through, or write-behind strategy between MongoDB and Redis. Justify your choice.
    Define your strategy for handling cache stampede (e.g., locking, request coalescing, jittered TTL).
    Discuss data consistency between the cache and the source of truth.
